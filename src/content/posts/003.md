React2Shell로 다시 생각해 본 시큐어 코딩의 중요성
개발을 공부하다 보면 처음에는 보안보다 기능 구현에 더 관심이 간다. 로그인 버튼을 눌렀을 때 로그인이 되는지, 게시글이 제대로 저장되는지, 화면이 원하는 대로 이동하는지 같은 것들이 먼저 눈에 들어온다. 나도 처음에는 에러 없이 돌아가면 어느 정도 완성된 코드라고 생각했다.
그런데 개발을 조금씩 하다 보면 “돌아가는 코드”와 “안전한 코드”는 다르다는 걸 느끼게 된다. 기능은 정상적으로 동작해도, 그 안에 보안상 허점이 있으면 사용자 정보가 유출되거나 서비스 전체가 공격당할 수 있다. 그래서 개발할 때는 기능뿐만 아니라 보안까지 같이 생각해야 한다. 이런 개발 방식을 보통 시큐어 코딩이라고 한다.
시큐어 코딩은 거창한 보안 기술이라기보다, 코드를 작성할 때 위험할 수 있는 부분을 미리 생각하고 줄여 나가는 습관에 가깝다.
보안 문제는 생각보다 가까이에 있다
보안이라고 하면 처음에는 굉장히 어렵고 먼 이야기처럼 느껴진다. 해킹, 서버 침투, 개인정보 유출 같은 단어가 먼저 떠오르기 때문이다. 하지만 실제로 많은 취약점은 아주 기본적인 부분에서 시작된다.
사용자가 입력한 값을 제대로 검사하지 않거나, 권한 확인을 빼먹거나, 외부 라이브러리 업데이트를 미루는 것처럼 사소해 보이는 부분이 문제가 될 수 있다. 개발자는 기능이 잘 되는지만 확인하고 넘어가기 쉽지만, 공격자는 그 빈틈을 찾는다.
그래서 시큐어 코딩의 첫 번째 출발점은 “내 코드가 어떻게 악용될 수 있을까?”를 한 번 더 생각해 보는 것이다.
사용자의 입력값은 믿지 않는 게 기본이다
웹 서비스에는 사용자가 값을 입력하는 부분이 많다. 로그인, 회원가입, 게시글 작성, 댓글, 검색창, 파일 업로드 등이 모두 여기에 해당한다. 겉으로 보기에는 평범한 입력칸이지만, 공격자는 이곳에 예상하지 못한 값을 넣을 수 있다.
예를 들어 나이를 입력받는 코드가 있다고 해보자.
const age = req.body.age;if (age >= 14) {  console.log("가입 가능");}
간단한 코드지만, 사용자가 항상 숫자만 보낸다는 보장은 없다. 문자열을 보낼 수도 있고, 음수나 비정상적으로 큰 값을 보낼 수도 있다. 그래서 서버에서는 입력값을 다시 확인해야 한다.
const age = Number(req.body.age);if (!Number.isInteger(age) || age < 0 || age > 120) {  return res.status(400).json({ message: "올바른 나이를 입력하세요." });}
프론트엔드에서도 입력값 검사를 할 수 있지만 그것만으로는 부족하다. 브라우저에서 하는 검사는 사용자가 마음만 먹으면 우회할 수 있다. 결국 중요한 검증은 서버에서 해야 한다. 프론트엔드 검증은 사용자 편의를 위한 것이고, 서버 검증은 보안을 위한 것이라고 생각하면 이해하기 쉽다.
SQL Injection은 오래됐지만 여전히 위험하다
SQL Injection은 대표적인 웹 취약점이다. 사용자가 입력한 값을 SQL 문장에 그대로 붙였을 때, 공격자가 쿼리의 의미를 바꿀 수 있는 문제다.
예를 들어 로그인 기능을 만들면서 아래처럼 쿼리를 작성하면 위험하다.
const query =  "SELECT * FROM users WHERE id = '" +  userId +  "' AND password = '" +  password +  "'";
이 방식은 입력값이 SQL 문장 안에 그대로 들어간다. 사용자가 정상적인 아이디와 비밀번호를 입력한다면 문제없이 보일 수 있지만, 공격자가 SQL 문법을 포함한 값을 넣으면 원래 의도와 다른 쿼리가 실행될 수 있다.
그래서 SQL을 작성할 때는 문자열을 직접 이어 붙이는 방식보다 파라미터 바인딩을 사용하는 것이 안전하다.
const query = "SELECT * FROM users WHERE id = ? AND password = ?";db.execute(query, [userId, password]);
이렇게 작성하면 사용자가 입력한 값은 SQL 명령어가 아니라 데이터로 처리된다. 요즘은 ORM을 많이 사용하지만, 직접 쿼리를 작성하는 부분에서는 여전히 조심해야 한다.
React를 사용해도 XSS를 완전히 피할 수는 없다
프론트엔드에서 자주 나오는 취약점 중 하나가 XSS다. XSS는 사용자가 입력한 악성 스크립트가 다른 사용자의 브라우저에서 실행되는 문제다. 게시판, 댓글, 프로필 소개처럼 사용자가 작성한 내용을 화면에 보여주는 기능에서 발생할 수 있다.
React는 기본적으로 JSX 안에 문자열을 출력할 때 이스케이프 처리를 해준다.
function Comment({ content }) {  return <p>{content}</p>;}
이렇게 작성하면 사용자가 입력한 값이 HTML 태그로 실행되지 않고 일반 텍스트처럼 출력된다. 그래서 React를 사용하면 기본적인 XSS 위험을 줄일 수 있다.
하지만 React를 쓴다고 해서 무조건 안전한 것은 아니다. 예를 들어 HTML을 직접 삽입하는 기능을 사용하면 위험해질 수 있다.
function Comment({ content }) {  return <div dangerouslySetInnerHTML={{ __html: content }} />;}
이 코드는 이름부터 위험하다는 느낌을 준다. 실제로 사용자가 입력한 HTML을 그대로 넣으면 악성 스크립트가 실행될 가능성이 생긴다. 정말 HTML을 넣어야 하는 상황이라면 먼저 내용을 정제해야 한다.
import DOMPurify from "dompurify";function Comment({ content }) {  const cleanHTML = DOMPurify.sanitize(content);  return <div dangerouslySetInnerHTML={{ __html: cleanHTML }} />;}
여기서 중요한 점은 프레임워크가 보안을 어느 정도 도와줄 수는 있지만, 개발자가 위험한 기능을 잘못 사용하면 취약점은 얼마든지 생길 수 있다는 것이다.
React2Shell 사례가 보여준 것
최근 React 생태계에서 많이 언급된 사례 중 하나가 React2Shell이다. React2Shell은 React Server Components와 관련된 취약점으로 알려졌고, 심각도가 높은 문제로 다뤄졌다. 이 사례를 보면서 시큐어 코딩이 단순히 내가 작성한 코드만 조심하는 문제가 아니라는 생각이 들었다.
예전에는 보안이라고 하면 주로 백엔드에서 처리하는 로그인, 비밀번호, 데이터베이스 정도를 떠올렸다. 그런데 React2Shell 같은 사례를 보면 프론트엔드 프레임워크도 서버와 연결되는 순간 보안 문제가 생길 수 있다.
요즘 React나 Next.js 같은 프레임워크는 단순히 화면만 그리는 도구가 아니다. 서버 사이드 렌더링, 서버 컴포넌트, API 라우트처럼 서버에서 동작하는 기능이 많아졌다. 그래서 프론트엔드 개발자도 서버 보안과 의존성 관리에 관심을 가져야 한다.
React2Shell이 특히 인상적인 이유는, 유명하고 많은 사람이 사용하는 기술에서도 취약점이 발견될 수 있다는 점이다. “React니까 괜찮겠지”라고 생각할 수 있지만, 실제로는 사용하는 버전과 설정에 따라 위험해질 수 있다. 결국 개발자는 자신이 사용하는 라이브러리와 프레임워크의 보안 이슈를 계속 확인해야 한다.
의존성 관리는 귀찮지만 꼭 필요하다
시큐어 코딩이라고 하면 보통 내가 직접 작성하는 코드만 떠올리기 쉽다. 하지만 실제 프로젝트에서는 외부 라이브러리를 정말 많이 사용한다. npm으로 패키지를 설치하고, 프레임워크가 제공하는 기능을 가져다 쓰고, 여러 도구에 의존해서 서비스를 만든다.
문제는 이 의존성 중 하나에 취약점이 있으면 내 서비스도 영향을 받을 수 있다는 것이다. React2Shell 같은 사례가 바로 그런 부분을 잘 보여준다.
프로젝트에서 어떤 패키지를 사용하고 있는지는 아래처럼 확인할 수 있다.
npm list react react-dom next
또는 package.json을 직접 확인해도 된다.
{  "dependencies": {    "react": "19.2.0",    "react-dom": "19.2.0",    "next": "15.0.0"  }}
취약한 버전을 사용하고 있다면 패치된 버전으로 업데이트해야 한다.
npm updatenpm audit
물론 업데이트가 항상 간단한 일은 아니다. 버전을 올렸더니 기존 코드가 깨지는 경우도 있고, 테스트를 다시 해야 할 수도 있다. 그래도 보안 취약점이 알려진 버전을 계속 사용하는 것은 더 위험하다. 기능 개발만큼이나 패키지 관리도 개발자의 책임이라고 생각한다.
인증과 인가는 따로 봐야 한다
보안에서 자주 헷갈리는 개념이 인증과 인가다. 인증은 사용자가 누구인지 확인하는 과정이고, 인가는 그 사용자가 어떤 행동을 할 수 있는지 확인하는 과정이다.
예를 들어 로그인을 했다고 해서 모든 게시글을 삭제할 수 있으면 안 된다. 일반 사용자는 자신의 글만 삭제할 수 있어야 하고, 관리자는 정책에 따라 더 넓은 권한을 가질 수 있다.
아래와 같은 게시글 삭제 API는 위험하다.
app.delete("/posts/:id", async (req, res) => {  await deletePost(req.params.id);  res.json({ message: "삭제 완료" });});
이 코드는 누가 요청했는지 확인하지 않는다. 실제 서비스라면 누군가 다른 사람의 게시글을 삭제할 수도 있다.
조금 더 안전하게 작성하면 다음과 같다.
app.delete("/posts/:id", async (req, res) => {  const post = await findPostById(req.params.id);  if (!req.user) {    return res.status(401).json({ message: "로그인이 필요합니다." });  }  if (post.authorId !== req.user.id && req.user.role !== "admin") {    return res.status(403).json({ message: "삭제 권한이 없습니다." });  }  await deletePost(req.params.id);  res.json({ message: "삭제 완료" });});
이처럼 로그인 여부뿐만 아니라, 해당 사용자가 그 작업을 할 권한이 있는지도 확인해야 한다. 실제 서비스를 만들 때 이런 검사를 빼먹으면 큰 문제가 될 수 있다.
비밀번호는 그대로 저장하면 안 된다
회원가입 기능을 만들 때 비밀번호 저장 방식도 중요하다. 비밀번호를 데이터베이스에 그대로 저장하는 것은 매우 위험하다.
await db.execute("INSERT INTO users (email, password) VALUES (?, ?)", [  email,  password,]);
만약 데이터베이스가 유출되면 사용자의 비밀번호가 그대로 노출된다. 그래서 비밀번호는 해시 처리해서 저장해야 한다.
import bcrypt from "bcrypt";const hashedPassword = await bcrypt.hash(password, 10);await db.execute("INSERT INTO users (email, password) VALUES (?, ?)", [  email,  hashedPassword,]);
로그인할 때는 사용자가 입력한 비밀번호와 저장된 해시값을 비교한다.
const isValid = await bcrypt.compare(inputPassword, user.password);if (!isValid) {  return res.status(401).json({ message: "로그인 정보가 올바르지 않습니다." });}
비밀번호 저장은 기본적인 내용처럼 보이지만, 실제로 실수하면 피해가 크다. 그래서 회원 기능을 만들 때는 반드시 신경 써야 하는 부분이다.
에러 메시지도 공격자에게 힌트가 될 수 있다
개발할 때는 자세한 에러 메시지가 도움이 된다. 어디에서 문제가 생겼는지 바로 알 수 있기 때문이다. 하지만 운영 환경에서 내부 에러가 그대로 사용자에게 보이면 문제가 될 수 있다.
app.use((err, req, res, next) => {  res.status(500).json({    message: err.message,    stack: err.stack,  });});
이런 코드는 서버 내부 경로나 구조가 노출될 수 있다. 공격자는 이런 정보를 보고 어떤 기술을 사용하는지, 어디를 노려야 할지 추측할 수 있다.
운영 환경에서는 사용자에게는 간단한 메시지만 보여주고, 자세한 내용은 서버 로그에 남기는 방식이 좋다.
app.use((err, req, res, next) => {  console.error(err);  res.status(500).json({    message: "요청을 처리하는 중 문제가 발생했습니다.",  });});
다만 로그에도 민감한 정보가 들어가면 안 된다. 비밀번호, 인증 토큰, 개인정보 같은 값이 로그에 남으면 로그 파일 자체가 또 다른 위험이 될 수 있다.
시큐어 코딩은 결국 습관이다
시큐어 코딩은 모든 보안 기술을 완벽하게 외우는 것이 아니다. 개발할 때 계속 한 번 더 의심하고 확인하는 습관이라고 생각한다.
이 입력값이 이상하게 들어오면 어떻게 될지, 이 API를 아무나 호출할 수 있지는 않은지, 이 패키지 버전은 안전한지, 에러 메시지가 너무 많은 정보를 보여주지는 않는지 확인하는 과정이 필요하다.
React2Shell 사례를 보면서 특히 느낀 점은 보안이 백엔드만의 문제가 아니라는 것이다. 프론트엔드에서도 서버와 연결되는 기능이 많아졌고, 프레임워크 내부 취약점이 서비스에 영향을 줄 수 있다. 그래서 프론트엔드 개발자도 보안 공지와 패키지 업데이트를 무시하면 안 된다.
마무리
시큐어 코딩은 어렵게 들리지만, 결국 작은 습관에서 시작된다. 입력값을 검증하고, SQL을 안전하게 작성하고, XSS가 생길 수 있는 부분을 조심하고, 권한 검사를 빼먹지 않고, 비밀번호를 안전하게 저장하고, 의존성을 꾸준히 관리하는 것부터 시작하면 된다.
완벽한 보안은 현실적으로 쉽지 않다. 그래도 개발자가 코드를 작성할 때 보안을 조금만 더 의식하면 많은 위험을 줄일 수 있다. 기능이 잘 돌아가는 것도 중요하지만, 사용자가 안심하고 사용할 수 있는 서비스를 만드는 것도 개발자의 중요한 역할이라고 생각한다.
